ECC Memory

Error-Correcting Code (ECC) memory adds hardware circuitry to detect and correct bit errors — cosmic ray strikes, electrical noise, and alpha-particle events that can flip a stored bit from 0 to 1 or vice versa.

How ECC Works

Standard ECC uses a SECDED (Single Error Correct, Double Error Detect) code — specifically a Hamming code variant. For every 64 bits of data, 8 check bits are computed and stored, forming a 72-bit ECC word.

  • Single-bit errors: Detected and automatically corrected on every read
  • Double-bit errors: Detected but not correctable — system logs an uncorrectable error (UCE)
  • 3+ bit errors: May go undetected

DDR5 On-Die ECC All DDR5 devices are required to implement on-die ECC. This corrects errors within the DRAM die before they reach the bus — but it is not a replacement for system-level ECC. ODECC errors are invisible to the memory controller and do not trigger OS-level error reporting.

Rowhammer

Rowhammer is a hardware vulnerability discovered by Kim et al. (CMU/Intel, 2014) where repeatedly accessing ("hammering") a DRAM row causes bit flips in physically adjacent rows — without ever directly accessing those rows.

The Physics

Every time a row is activated, the word line voltage toggles. This couples electromagnetic noise into adjacent bit lines. Over thousands of activations, the disturbance charge accumulates in nearby cells, eventually flipping bits stored there. Modern DRAM is especially vulnerable because cells are packed closer together at each process node.

Rowhammer — Aggressor and Victim Rows
Row N-2 (safe)
Row N-1 — VICTIM ⚠ (bit flips here)
Row N — AGGRESSOR (hammered 10,000s of times)
Row N+1 — VICTIM ⚠ (bit flips here)
Row N+2 (safe)

Security Impact

Rowhammer can be exploited to escalate privileges, escape sandboxes, and break cryptographic isolation — all without any software vulnerability. Notable attacks include:

  • rowhammer.js (2016): Exploited from a browser using JavaScript (no native code required)
  • Drammer (2016): Rowhammer privilege escalation on Android
  • TRRespass (2020): Bypassed Target Row Refresh (TRR) mitigations via many-sided hammering
  • Blacksmith (2021): Non-uniform hammer patterns that bypass all known TRR implementations
  • RowPress (2023): Bit flips from keeping a row active for extended time — not just toggling

Mitigations

MitigationHow It WorksEffectiveness
TRR (Target Row Refresh)DRAM internally refreshes neighbors when a row is frequently activatedBypassed by TRRespass/Blacksmith
PARA (Probabilistic Adjacent Row Activation)Memory controller randomly refreshes neighbors after activationsGood; proposed in academic literature
pTRR / RFM (Refresh Management, DDR5)DRAM signals controller when a row's activation count is high; controller issues targeted refreshesStrong; mandated in DDR5

Still an Active Threat As of 2026, Rowhammer and its descendants remain practically exploitable on many deployed systems. DDR5 with RFM provides the strongest mitigation to date, but older DDR4 systems remain exposed. Cloud providers and OS vendors continue to patch isolation boundaries.

3D Stacking & Through-Silicon Vias

HBM (covered in Module 03) is the best-known application of 3D die stacking, but the underlying technology has broader implications for DRAM scaling.

Through-Silicon Vias (TSVs)

A TSV is a vertical electrical connection drilled through a silicon die and filled with copper. It allows signals and power to pass between stacked dies without long wire bonds. TSVs enable:

  • 1024-bit wide bus between HBM dies and the base logic die
  • Very short interconnects → lower capacitance → lower power per bit transferred
  • Dense stacking: up to 12 DRAM dies in a single HBM3E stack

2.5D vs 3D Integration

2.5D (Interposer)

  • DRAM stack and logic die sit side-by-side on a silicon interposer
  • Connected via short interposer traces (not TSVs between logic and DRAM)
  • Used in: HBM-based GPUs (AMD MI300, NVIDIA H100)
  • Simpler integration; known-good die testing before assembly

3D (Direct Stacking)

  • DRAM dies stacked directly on top of logic die using TSVs
  • Shortest possible interconnect — lowest latency, highest bandwidth density
  • Used in: AMD 3D V-Cache (SRAM), prototype DRAM-on-logic chips
  • Higher risk: failed DRAM die kills the whole package

Processing-in-Memory (PIM / NDP)

The fundamental bottleneck in data-intensive computing is moving data from DRAM to the CPU — the "memory wall." Processing-in-Memory (PIM) and Near-Data Processing (NDP) move compute closer to, or inside, the DRAM itself.

Approaches

  • HBM-PIM (Samsung Aquabolt-XL): SIMD compute units integrated into each HBM pseudo-channel. Can execute simple operations (vector addition, activation functions) without moving data off the package.
  • UPMEM PIM DIMMs: Standard DDR4 DIMMs with 8 RISC cores per DRAM chip — 2500+ cores per module. Used for genomics, database, and graph analytics.
  • Micron GDDR6-AiM: AI-in-Memory GDDR6 with dot-product accelerators adjacent to each bank.
  • CXL Memory Devices with Compute: CXL 3.x enables active memory devices with on-board processing accessible via standard protocol.
The Memory Wall CPU compute throughput has grown ~3.5× per decade; DRAM bandwidth has grown ~2×. The gap widens every generation. For workloads that are bandwidth-bound (most ML inference, genomics, graph processing), moving compute to the data is more efficient than the reverse.

CXL — Compute Express Link Memory

CXL (Compute Express Link) is a cache-coherent interconnect standard built on top of PCIe 5.0/6.0 physical layer. CXL.mem allows a host CPU to access memory on an attached CXL device with cache-coherent semantics — the CPU can load/store to CXL memory just like DRAM, and the hardware maintains coherency automatically.

CXL enables memory disaggregation: instead of each server having dedicated DRAM, a datacenter can pool memory resources and allocate them to workloads on demand. This improves utilization and reduces over-provisioning costs.

Reliability & Failure Modes

DRAM in production systems faces multiple reliability challenges beyond Rowhammer:

Retention Failures

At elevated temperatures or high cell leakage (manufacturing variation), some cells drain faster than others. If tREFI is too long, these weak cells lose data between refreshes. This is why servers operate at the conservative JEDEC tREFI and why high-temperature operation requires more frequent refresh.

Variable Retention Time (VRT)

Some cells exhibit metastability — their retention time randomly switches between fast-leaking and normal-leaking states on a timescale of minutes to hours. This makes them nearly impossible to detect with standard burn-in tests, and they can cause field failures in production.

Write Disturb

Writing to one cell can disturb the stored charge in physically adjacent cells, similar to (but distinct from) Rowhammer. More significant in high-density 3D NAND, but present to a lesser degree in DRAM at advanced nodes.

Cosmic Ray Single-Event Upsets (SEU)

High-energy cosmic ray neutrons can strike a cell and deposit enough charge to flip its state. At sea level, the soft error rate for DRAM is approximately 1 FIT per 1 Mb (1 failure per 10⁹ hours per megabit). A 256 GB server has ~2 trillion bits — statistically, a cosmic-ray bit flip every few hours without ECC.

Beyond DRAM — What Comes Next?

DRAM scaling is approaching fundamental physical limits. The 1T1C cell requires a minimum capacitor size to hold enough charge, and the access transistor cannot shrink indefinitely without excessive leakage. The industry is exploring several post-DRAM technologies:

TechnologyPrincipleStatusAdvantage over DRAM
MRAM (Magnetic RAM)Magnetic tunnel junction — spin state stores dataProduction (embedded); GB-scale in developmentNon-volatile, no refresh, radiation-hard
FeRAM (Ferroelectric RAM)Polarization of ferroelectric capacitor stores bitNiche production (TI, Fujitsu)Non-volatile, low power write
PCM (Phase-Change Memory)Crystalline vs. amorphous resistance differenceProduction (Intel Optane — discontinued 2022)Non-volatile, higher density potential
RRAM / ReRAMFilament formation changes resistanceResearch; limited productionSub-5 nm scaling potential
2T0C DRAMTwo transistors, no capacitor — charge in floating bodyResearch (Toshiba, others)Better scaling without capacitor
3D DRAMVertical cell arrays — grow upward instead of laterallyEarly development (Samsung, SK Hynix)Density scaling continues past 2D limits
DRAM Will Survive Despite decades of predictions that DRAM would be replaced, it has outlasted every challenger. Its combination of density, cost, performance, and mature manufacturing ecosystem is extraordinarily difficult to displace. The most likely near-term scenario is DRAM co-existing with emerging NVM in a multi-tier hierarchy — DRAM for working set, persistent memory for capacity — connected via CXL.
